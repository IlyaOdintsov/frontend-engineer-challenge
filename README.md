# Frontend Engineer Challenge

Frontend проект auth-флоу под реальный [backend fork](https://github.com/aiezq/engineer-challenge)

### Локальный запуск

```bash
# Backend
Запустить [backend](https://github.com/aiezq/engineer-challenge)

# Frontend
yarn install
yarn dev

# Тесты 
yarn test
```

## Архитектура frontend

Проект построен по Feature‑Slice‑подобной архитектуре с разделением на слои:

- `src/app` — точка входа, AppRouter, глобальные провайдеры.
- `src/entities` — доменные сущности и типы
- `src/features` — бизнес‑фичи: `auth-login`, `auth-register`, `forgot-password`, `auth-layout`.
- `src/pages` — основные страницы, которые используют нужные features и layout.
- `src/shared` — общие слои:
    - `api` — централизованный GraphQL‑клиент.
    - `ui` — переиспользуемые компоненты (FormInput, Button, BackButton, Layout, Footer).
    - `store` — Zustand‑хранилище (authStore).
    - `hooks` — общие хуки: `useSubmitForm`.
    - `lib` — утилиты и хелперы (errors, `AuthGuard`, `ProtectedRoute`).
- `src/widgets` — зарезервированный слой для будущих вертикальных блоков (микрофронтенд‑подобных компонентов), пока пуст.

## Backend
Выбранный backend — это GraphQL‑API‑сервер, который предоставляет следующие операции:

- register(input: RegisterUserInput!) — регистрация пользователя.

- authenticate(input: AuthenticateInput!) — аутентификация.

- requestPasswordReset(input: RequestResetInput!) — запрос сброса пароля (отправка письма).

- resetPassword(input: ResetPasswordPasswordInput!) — смена пароля по токену.

Этот fork реализует стандартный auth‑флоу (register/login/forgot password/reset), 
использует GraphQL‑мутации для всех действия, а не REST‑эндпоинты,
возвращает структурированные data + errors, что удобно типизировать фронтендом.

На фронтенде все запросы к API проходят через центральный authApi‑слой:

- для register, login, requestPasswordReset, resetPassword вызываются GraphQL‑мутации с соответствующими input‑объектами;


- ответы типизируются как ApiResponse<T>, где T — соответствующий GraphQL‑тип;


- ошибки из errors мапятся через shared/lib/errors.ts в понятные пользователем сообщения.

Почему был выбран именно GraphQL‑backend, а не REST‑аналог:

- GraphQL позволяет запрашивать ровно те поля, которые нужны, а не весь user‑объект.


- Типизация на стороне сервера хорошо сочетается с TypeScript‑клиентом и Zod‑валидацией.


- Мутации дают единый контракт для всех auth‑действий, что упрощает написание обобщённых useSubmitForm‑хуков.

## Стек
### React + React Router
React — популярный и понятный фреймворк, с большим количеством готовых решений и примеров.
React Router 7 позволяет легко организовать переходы между страницами авторизации, регистрации и восстановления пароля, не усложняя код.

Альтернативы:

- Next.js — удобный, но избыточный для простого auth‑флоу, где не нужны SSR/SSG и роутинг на уровне файлов.

- Vue + Vue Router — тоже хороший вариант, но в React‑ориентированной команде он будет чуть менее привычен.

### React Hook Form
React Hook Form даёт легковесный и удобный способ работать с формами, сохраняет значения в ref, 
почти не перерисовывает лишнее, хорошо сочетается с Zod и делает валидацию форм простой и понятной,

### Zod
Zod используется для валидации форм и схем данных:
валидация работает как на уровне формы, так и на уровне входных данных перед отправкой на бэкенд.

Альтернативы:

- Yup — более старый, но менее удобный в TypeScript и с большим количеством boilerplate.

- собственная валидация через ручные if/else — проще, но сложнее поддерживать и масштабировать.

### GraphQL (graphql-request)
GraphQL позволяет запрашивать только нужные поля и объединять все действия по авторизации (регистрация, вход, сброс пароля) в мутации, а не в отдельные REST‑эндпоинты.

Альтернативы:

- REST API (axios + обычные POST /auth/login) — привычный, но менее гибкий и не даёт такой удобной типизации по полям.

- Apollo Client — мощный, но с большим количеством настроек, что избыточно для auth‑флоу.

### Zustand
Zustand — лёгкий стейт‑менеджер, который не требует сложной структуры store и лишнего boilerplate.

Альтернативы:

- Redux — мощный, но более тяжёлый и сложный, особенно для такой задачи.

- Context + useReducer — ручной вариант, требует больше усилий по организации и масштабированию.

### Tailwind CSS + Vite
Tailwind CSS — удобный low‑level CSS‑фреймворк, который позволяет быстро писать адаптивный UI, не придумывая кучу собственных классов.

Альтернативы:

- SCSS/LESS — классический препроцессор, но требует больше ручной работы с классами и структурой.

- styled-components / emotion — CSS‑in‑JS, но Tailwind проще и легче масштабировать для такого объёма.

### MSW + Vitest + @testing‑library/react
MSW мокает HTTP‑запросы, включая graphql-request, чтобы тестировать формы, ошибки и успешные сценарии без реального бэкенда.
Vitest — быстрый тестовый фреймворк, который хорошо интегрируется с Vite и React.
@testing‑library/react помогает писать понятные, user‑ориентированные тесты для форм и кнопок.

Альтернативы:

- Jest — стандартный, но чуть медленнее и сложнее в настройке.

- Cypress / Playwright — больше подходят для e2e, как дополнение, а не замена unit/integration‑тестов.

## Trade‑offs
   Выбранный стек и архитектура — это компромисс, а не идеальный набор. Ниже — ключевые trade‑offs, которые я осознанно сделал и почему так решил.

1. Я не внедряю полноценную дизайн‑систему (Storybook, tokens и т.п.),
 добавление Storybook, дизайн‑токенов и полной библиотеки компонентов было бы 
избыточно по объёму, но я уже выделил готовый слой для будущей дизайн‑системы.


2. Я не использую полностью FSD‑архитектуру, но структура уже близка к этому подходу (entities, features, shared, pages, app).
FSD — это мощный, но достаточно тяжёлый паттерн, рассчитанный на более крупные приложения, поэтому для 
задания auth‑флоу я выбрал упрощённую, feature‑slice‑подобную структуру.


3. Я не использую микрофронтенды, так как проект малый и не нуждается в такой сложности.
Микрофронтенды оправданы, когда есть несколько независимых команд и отдельных библиотек, которые нужно интегрировать,
поэтому вводить разделение на самостоятельные приложения/фичи было бы усложнением.


4. Я не использую Redux, а выбрал Zustand, так как для auth‑флоу он достаточно лёгкий и удобный.
Redux требует отдельного store, actions, reducers, connect и т.п., что избыточно для простого состояния авторизации.
Zustand даёт лёгкий, понятный и типизируемый доступ к authStore без лишней инфраструктуры, что полностью покрывает текущие потребности проекта.
